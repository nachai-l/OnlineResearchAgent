## Decisions worth remembering

- **Selection as its own module.** Plan listed the prompt but not
  the module. Splitting it out mirrors the `validation.py` /
  `summarization.py` pattern and made the pipeline trivially
  testable as pure orchestration.
- **Event stream stays coarse.** Chose `working → artifact →
  completed` at the A2A layer rather than emitting one status per
  internal stage. Internal stages already log structured lines;
  re-emitting them as A2A events would flood clients with detail
  they can't act on. If a future client wants fine-grained progress
  we can add it as an opt-in extension.
- **Agent Card URL is built once at server-startup.** The card
  reflects the configured `server.host:port`. Dynamic card
  modifiers are an SDK feature but not needed at this scale.
- **Wiring is an integration seam, not new logic.** Intentionally
  not unit-tested — its only job is to pass the right fields to
  already-tested constructors. If a bug lands there it surfaces in
  the live smoke.
- **`sse-starlette` is a hard dep for the server.** Added
  `a2a-sdk[http-server]` + explicit `sse-starlette` to
  `requirements.txt` so a fresh checkout installs everything.

## Addendum: LiteLLM backend + Docker deployment

After the live smoke shook out (key-leak fix in `utils/logging.py`,
zombie-cache auto-heal in `core/scraping.py`, full missing-deps
install), the agent was asked to (a) talk to an internal LiteLLM proxy
instead of Gemini directly, and (b) deploy as a container.

### LiteLLM as a second LLM backend

Already had the hard work done: `llm/client.py` exposes a `CallFn`
Protocol and the production path goes through `default_call_fn`. So
the integration is purely additive.

- **New**: `default_litellm_call_fn(*, base_url, api_key)` in
  `functions/llm/client.py`. POSTs canonical OpenAI chat-completions
  (`{"model", "messages": [system, user], "temperature",
  "max_tokens"}`) to `{base_url}/chat/completions` with
  `Authorization: Bearer ...`, reads `choices[0].message.content`.
  Trailing slashes on `base_url` are tolerated. Empty `choices`
  raises `LlmCallError` instead of returning an empty string so the
  retry envelope can catch it.
- **Timeout**: `httpx` alone wasn't enough — respx mocks skip the
  transport-layer timeout. Wrapped the request in
  `asyncio.wait_for(..., timeout=timeout)` so the contract holds
  against any transport and the timeout test is meaningful.
- **Tests**: 5/5 (`tests/test_llm_litellm.py`) — OpenAI shape +
  Authorization header, trailing-slash base_url, non-2xx → HTTPStatusError,
  empty choices → LlmCallError, timeout propagation.

Backend selection is a config flag, not code:

- `LlmSettings.backend: Literal["gemini", "litellm"]` (default
  `"gemini"`, switched to `"litellm"` in `parameters.yaml`).
- `LlmSettings.litellm_base_url: str | None` — only consulted when
  backend is `"litellm"`.
- `Credentials.litellm_api_key: str | None` — resolved from
  `LITELLM_API_KEY` via the existing env-var-pointer pattern in
  `credentials.yaml`.
- `functions/agent/wiring.py` grew a private `_build_call_fn` that
  picks the right factory and raises a clear error at startup if the
  wrong credentials are missing, rather than lazy-failing on the
  first LLM request. `GeminiClient` is now always constructed with
  `call_fn=<the chosen fn>` and `api_key=None` — the legacy
  "build-SDK-fn-on-demand" path in the client still works but the
  wiring no longer relies on it.

Key design intent: the name `GeminiClient` is now a misnomer (it's
just "the LLM client with cache + retry + JSON parsing") but I didn't
rename it. That's a cosmetic change worth one PR on its own; squeezing
it into this one would have made the diff noisy.

### Docker deployment

Three new files at the repo root. No source touched except
`scripts/run_server.py`.

- **`Dockerfile`** — two-stage build on `python:3.12-slim`.
  - `builder` stage has `build-essential` + `libxml2-dev` +
    `libxslt1-dev` so trafilatura's `lxml` wheel compiles cleanly.
    Deps install into `/opt/venv`; nothing else from the stage
    survives into the runtime image.
  - `runtime` stage pulls in runtime-only `libxml2` + `libxslt1.1`,
    copies the venv + source tree (`functions`, `prompts`, `configs`,
    `scripts`), creates `/app/artifacts/{cache,logs}`, and switches
    to a non-root `app` user. Defaults
    `A2A_HOST=0.0.0.0 A2A_PORT=8000`.
  - `HEALTHCHECK` hits
    `/.well-known/agent-card.json` via `urllib` — no extra dep.
- **`.dockerignore`** — excludes `.env*`, `.venv/`, caches, tests,
  docs, `.git/`. The image is source + deps, nothing else.
- **`docker-compose.yml`** — one service. Reads `.env` via
  `env_file`. Binds host port `${A2A_HOST_PORT:-8000}` →
  container 8000. Bind-mounts `./artifacts/cache` and
  `./artifacts/logs` so JSONL caches and logs survive
  `docker compose down` and are greppable from the host.
  `A2A_PUBLIC_URL` defaults to `http://localhost:8000` and can be
  overridden for reverse-proxy deployments.

`scripts/run_server.py` now reads three optional env-var overrides:

- `A2A_HOST` — bind address (defaults to `parameters.yaml`).
- `A2A_PORT` — bind port.
- `A2A_PUBLIC_URL` — URL advertised in the Agent Card.

That last one matters: inside a container the app binds `0.0.0.0:8000`,
but clients need the externally reachable URL
(`http://localhost:8000` locally, `https://...` behind a proxy).
Without `A2A_PUBLIC_URL` the card would advertise `0.0.0.0:8000` which
is never what a client wants.
